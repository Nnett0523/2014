regex

Regular expressions can be used almost anyywhere, in almost all environments and programming languages.

This is because they are so useful.

For example a good regex can be equivalent to a many-lines program, with the additional advantage of being much leaner and thus, much easier to test by inspection.

A regex defines a language, usualy a small one. This is easy to accept if we consider the language's BNF definition and realize that, in fact, it's equivalent to a regex.

So, if the regex can express a small language, then running it over a piece of text is comparable to parsing the text for compliance, and tokenizing it.

As a result, the regex replaces a full parser program.

Generalities

A small set of concepts needed for full understanding of the subject.

How does it looks like?

A regex example

A Linux user is listing a lengthy directory containing hundreds of files. She enters the ls -l command (or dir if in Windows) and the command floods the screen with the files listing.

She is interested only in the .xls or .xlsx or .xlsm Excel files containing the year "2014" somewhere in the name.

Instead of reading the many entries and picking the ones she's interested in (risking to miss a few), she feeds the listing into a grep command furnished with a regex that implements her micro "language".

Her "language" consists, then, in the set of "words" that start with zero or more characters, contain the string "2014", then zero or more characters again, the string ".xls" and an optional extra "x" or "m".

The regex that matches such strings is .*2014.*\.xsl[xm]*

It is made up of several parts, let's analyze each one in isolation:

.* The dot means "any character" and the asterisk is a multiplier that means "zero or more times", so this matches all strings made of any characters, followed by ...
2014 the exact string "2014", followed by ...
.* any character zero or more times (again), plus ...
\.xsl a dot and the characters "xsl" - the dot was escaped here to cancel its magic so it matches only "." - and finally ...
[xm]* one of the two "x" or "m", zero or more times.

The filtered directory listing command ends up like so:ls -l | grep ".*2014.*\.xslx*"

For the who don't know it means "list the directory files in long format and pipe the listing to a grep command that will output only the lines containing a piece of text that matches the regex we know.

Notice that the filter is not perfect. It will match file names like a2014.xslxxxmmm with any number of "x"s and "m"s, and although it's quite possible to make the regex stricter, it's usually not needed because those file names are unlike to appear, and if one shows it's an opportunity to fix it.

Don't feel embarrassed because your PC runs Windows and your can't use commands like this. Install MINGW and you are done.
