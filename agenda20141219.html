How do we work, and why

the workflow

  1. We, as a sub-team, have a set of pending issues that we call "the backlog" following the agile terminology.
  2. I short meetings (like 5' long) we choose which issue will start each one of us, our local TL (Andrés) validates the choices.
  3. We are using Git on SVN (the git-svn feature) so we create (checkout in Git parlance) a new branch to isolate the forthcoming code changes to a potentially high number of files. The branch is given a significant number, after the Jira card, including its number. Having the changes isolated in a branch allows us to revert then in case we needed to do so, to see the detail in a diff tool, or to stash them locally in case we had half-baked changes and needed to switch to another task.
  4. Now comes the easiest part and the most rewarding, which is to do the coding so the issue gets solved. This part is depicted separately.
  5. When the developer thinks the code is ready, after having tested it, he asks for a code review. He asks another developer of the team for the review.

    The owner of the changes shows the reviewer the changes made, in great detail, using a diff tool (SVN has a good one).

    It is the responsibility of the reviewer to get to understand the changes in detail and to suggest changes if he sees that any good practices are vulnerated.

    The aurhor of the changes is asked to show and explain the code, to show how it works both in a browser and in IE8, and to show any other related files.

 

Step 4 - coding:

  1. The owner of the issue defines a strategy to solve it.
  2. The strategy is validated discussing it with the others, and eventually asking for definitions to other members of the whole team (local and remote), personally or via chat in order to have quick answers because these undefinitions are blocking.

 

Step 5 - code review:

Code reviewing is a QA (Quality Assurance) practice. QA strives to prevent defects, as opposed to QC (Quality Control) that detects defects after they happened. QA happens before while QC happens after. QA beats QC by leaps and bounds, and is's much less expensive because it doesn´t involve any rework but in the early development stages.

Look darling, I resized the children!

In web UI development there are may instances when the widths of a set of contained items have to be adjusted due to a change in the container's width.

available solutions

AFAIK there are these approaches:

  • Giving all children a percent width so that the sum of all widths is slightly less than 100%, and
  • Having only two elements, fixing one element's width and floating it, and setting the other elemtn's width to auto in order to create a block formatting context as in this example.
  • Bind the resize event to a JavaScript function that recalculates all the widths according to a given policy.

Both of these two methods ensure full tityness on resize.

The percents method is restrictive in that all the contained elements are resized proportionally and while this can be desirable in some cases, it won't in many others like for example one or more elements have fixed widths, minimum widths, or don't look good when widened beyond a number.

The BFC method is restrictive in that it works for two and only two elements. Yes, one can create nested BFCs, but it's cumbersome and still restrictive.

The third method, using a function, is the most versatile but on the downside it requires the function to be written, tested and mantained, which is overkill for many small instances of the problem.

generalizing the problem

The children resizing scenarios that I can think of are a combination of the following methods, where each child can cater from its preferred one:

  • fixed width
  • percent of the container's width (with optional max and min constraints)
  • proportional (same as percent only not 100-based)
  • percent of another's element width
  • largest text in a list
  • largest text in a table column (up to a limit)
  • all the remaining space

It seems possible to write a program implementing all these strategies, after getting enough information.

For example the resizing needs of a 3-elements set could be addressed by a jQuery plugin parameterized thus: 

$('#threeItemsContainer').tidyResize([
    {strategy:percent, width:25%, max-width:120px},
    {strategy:collct},
    {strategy:fixed, width:50px} 
]);
